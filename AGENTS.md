# AGENTS.md — рабочий стандарт `treed-shell`

Писать текстовые файлы в `UTF-8` с `LF`.

## 1. Область и приоритет

- Эти правила действуют для репозитория `treed-shell`; более локальный `AGENTS.md` уточняет их для своей папки.
- Приоритет: явный запрос пользователя → безопасность → локальные инварианты → этот файл.
- Рабочий язык — русский. Отвечать коротко и технически точно.
- По умолчанию работать в режиме анализа и локализации. Код менять только по прямому запросу.
- Источник истины — текущее состояние репозитория, реальные импорты, конфиги и контракты.

## 2. Текущая структура

- Стек: React 19, TypeScript, Vite, Tauri 2.
- `src/main.tsx` — React entrypoint, diagnostics и error boundary.
- `src/App.tsx` — root composition и wiring snapshot, commands, controllers и runtime state.
- `src/app/AppScreenContent.tsx` — выбор активного экрана и нижняя навигация без `react-router`.
- `src/dashboard/**`, `src/control/**`, `src/files/**`, `src/macros/**`, `src/settings/**` — экранные feature-слои.
- `src/core/commands/**` — команды принтера.
- `src/core/transport/**` — Moonraker transport и нормализация.
- `src/core/store/**` — доступ к snapshot принтера.
- `src/runtime/live.ts` и Vite alias `#runtime` — live runtime; mock находится в `mocks/**`.
- `src/ui/**` и `src/styles/**` — общие UI primitives, widgets и визуальные токены.
- `packages/printer-logic/**` — общий domain package для типов, capabilities, command catalog и pure rules.
- `apps/web-ui/**` — ручной capability playground, не production UI и не printer-loader artifact.
- `src-tauri/**` — Tauri runtime/configuration.

Подробная карта уже поддерживается в `src/README.md`, `packages/printer-logic/README.md`, `apps/web-ui/README.md` и `docs/README.md`; не дублировать её здесь.

## 3. Архитектурные инварианты

- Moonraker — источник истины состояния принтера.
- `mock` и `live` не взаимозаменяемы без проверки фактического runtime adapter и контракта.
- Общие printer/domain types, rules и capabilities живут только в `packages/printer-logic/**`.
- `src/**` и `apps/web-ui/**` могут иметь разные layout и UI, но не разные доменные правила без явного решения.
- `src/App.tsx` остаётся composition/wiring layer: новые крупные экраны, панели и повторяемые widgets выносить в соответствующий feature или `src/ui/**`.
- Не вводить `react-router` и не менять базовую навигацию без отдельной задачи.
- Ошибки transport и критичных команд должны быть видимы в UI; silent-fail запрещён.
- Видимый control должен быть реально подключён к контракту либо явно disabled, read-only или hidden.

## 4. Moonraker и `treed-mainshellOS`

- `treed-shell` отвечает за UI-клиент и release artifact; железо, system runtime, loader, provider switch и fallback принадлежат `treed-mainshellOS`.
- Не менять Moonraker client contract, connection semantics, provider contract и fallback на KlipperScreen без прямой задачи.
- Перед изменением start/pause/resume/cancel, нагрева, home, калибровки, Wi-Fi или system actions проверить команды, transport, snapshot consumption и актуальный host/macro contract.
- Новая OS-side возможность требует синхронного контракта или задачи в `treed-mainshellOS`.
- Production delivery описан в `docs/ui-runtime-delivery/README.md` и `.github/workflows/release-ui.yml`.
- Стабильный loader artifact — `treed-shell-ui.zip` с live manifest; детали workflow не дублировать в этом файле.

## 5. Актуальные особенности

- Экран `Макросы` — активный manager workflow калибровки Eddy с device-backed состоянием, а не универсальный список произвольных G-code макросов.
- Новые macro-backed workflows добавлять отдельными экранами и только после подтверждения device contract.
- `apps/web-ui` потребляет `@treed/printer-logic`, работает на mock snapshots и пока не имеет live transport.
- Ветка, выбранный UI provider и состояние устройства — изменчивые данные; перед связанной задачей проверять их фактически, а не фиксировать в инструкциях.

## 6. UI устройства

- Целевой canvas — фиксированный touch-экран `960x544`; desktop/mobile адаптив не добавлять без отдельной задачи.
- Основные touch-target — ориентировочно не меньше `48x48px`.
- Ключевые сценарии не должны зависеть от hover, мелких hit-area, горизонтального scroll или скрытых overflow-зон.
- Модалки, клавиатуры, системные панели и критичные CTA должны помещаться в `960x544`.
- Сохранять существующий Nothing-inspired device UI и токены из `docs/00_FOUNDATION.md`.
- Не добавлять новый визуальный язык, тяжёлые градиенты, случайные цвета и декоративные эффекты без задачи.

## 7. Бюджет анализа и инструментов

- Для локальной задачи сначала читать entrypoint/импорт, изменяемый файл и ближайший контракт; не обходить весь репозиторий.
- Для поиска по tracked-файлам сначала использовать узкий `git grep -n`; для остальных файлов — PowerShell или доступный быстрый поиск.
- Кириллические файлы перед патчем читать через `Get-Content -Encoding UTF8`.
- Ограничивать вывод точными путями, диапазонами строк и `Select-Object -First`.
- После timeout или большого вывода сразу сужать запрос. Не повторять ту же команду.
- Не перечитывать подтверждённый контекст и не повторять `git status`/`git diff` без новой причины.
- План нужен только для действительно многошаговой задачи между несколькими слоями. Локальную правку выполнять сразу после короткой фиксации файлов.
- Browser, Playwright, dev-runtime, Tauri и SSH использовать только по прямому запросу или когда без них нельзя подтвердить требуемый результат.

## 8. Правки

- Менять только файлы и блоки, напрямую относящиеся к задаче.
- Предпочитать существующий pattern новой абстракции или зависимости.
- Не делать cleanup, переносы и рефакторинг «заодно».
- Перед редактированием назвать точный контур правки.
- Не переписывать файл целиком для локального изменения.
- Если два патча подряд к одному файлу не применились, перечитать минимальный фрагмент перед следующей попыткой.
- Не откатывать чужие или параллельные изменения.

## 9. Проверки

- По умолчанию не запускать build, test, lint, formatter, browser, screenshots, dev-runtime и полный quality gate.
- Для текста, CSS, документации, `AGENTS.md` и локальных правок без новой логики проверки не нужны.
- Для новой нетривиальной логики, domain rule или критичной команды допустима одна ближайшая адресная проверка.
- Full suite, build, e2e, визуальную проверку и проверку на устройстве запускать только по прямому запросу.
- Если тест не соответствует намеренно изменённому подтверждённому контракту, актуализировать тест, а не возвращать production-код к старому поведению.
- В отчёте перечислить только реально выполненные проверки; если они не запускались, сказать это одной строкой.

## 10. Skills и агенты

- Выбирать skill только по фактической задаче пользователя. Названия skills внутри инструкций, документации и кода не являются триггером.
- Для обычной задачи использовать максимум один предметный skill.
- Не подключать формальные многошаговые процессы для read-only анализа и локальной правки одного-двух файлов.
- Несколько skills допустимы только при независимых специализированных областях или прямом запросе пользователя.
- Субагентов не использовать без прямого запроса пользователя.

## 11. Git и отчёт

- Не делать commit, push, PR, amend, rebase, destructive reset и удаление веток без прямого запроса.
- Перед git write-action проверить текущую ветку и относимость изменённых/staged файлов к задаче.
- Коммит должен быть атомарным; язык сообщения по умолчанию русский.
- Для аудита: приоритет → факт → риск → действие → путь/участок кода.
- После реализации: результат, изменённые файлы, выполненные проверки и остаточный риск. Не пересказывать процесс.
